# frontend-design + ui-ux-pro-max 协作模板

## 目的

让 `frontend-design` 负责“定风格与做出记忆点”，让 `ui-ux-pro-max` 负责“收口与专业化”，避免两者在同一轮里互相拉回平均值。

## 固定顺序

1. 先调用 `frontend-design`
2. 再调用 `ui-ux-pro-max`
3. 不要在第一轮同时要求两者一起直接生成最终代码

## 职责划分

### frontend-design 负责

- 视觉方向
- 目标受众气质
- 页面记忆点
- 版式骨架与主视觉语言
- 配色、字体、构图、材质、氛围

### ui-ux-pro-max 负责

- 信息层级
- 间距与布局系统
- 交互反馈与状态完整性
- 响应式
- 可访问性
- 一致性与设计系统化

## 越界限制

### frontend-design 不应主动覆盖

- 详细 a11y checklist
- 设计系统收口规则
- 过细的表单/状态规范

### ui-ux-pro-max 不应主动推翻

- 已确定的整体风格方向
- 主视觉概念
- 记忆点
- 已批准的核心字体与配色气质

若发现风格与可用性存在真实冲突，应只指出冲突点并给出局部修正，不要整页改回通用模板风格。

## 推荐工作流

### 第一轮：只给 frontend-design

目标：先产出强风格方向，不直接进入“稳妥但普通”的最终页。

可直接使用下面这段提示词：

```md
请只使用 frontend-design 的职责来工作，先不要做 ui-ux-pro-max 式的规范收口。

任务：为这个页面先定义高辨识度视觉方向，再给出页面骨架。

必须先明确：
- 产品/页面目的
- 目标受众
- 想传达的气质
- 不希望落入的常见风格
- 允许偏离现有项目风格的范围
- 参考物或灵感来源（如果有）

输出要求：
1. 给出 3 个明显不同的视觉方向
2. 每个方向包含：风格名、关键词、主色气质、字体建议、版式特点、记忆点
3. 明确推荐其中 1 个方向，并说明为什么
4. 基于该方向给出页面骨架与关键视觉设计点

此阶段不要把结果收缩成通用 SaaS 模板，也不要优先做可访问性/设计系统收口。
```

### 第二轮：再给 ui-ux-pro-max

目标：在不推翻第一轮风格的前提下，把页面做得更专业、可用、完整。

可直接使用下面这段提示词：

```md
现在进入 ui-ux-pro-max 阶段。

前置约束：frontend-design 已经确定了视觉方向、页面骨架、主要字体/配色气质与记忆点；这些内容默认视为已批准，不要整体推翻。

你的职责仅限于：
- 优化信息层级
- 补齐交互状态
- 调整间距与布局系统
- 提升响应式表现
- 检查可访问性与一致性

如果发现前一阶段的风格选择与可用性冲突：
1. 只指出具体冲突点
2. 提供局部修正方案
3. 不要把整体页面改回保守、普通、模板化的风格

输出要求：
1. 列出需要收口的关键点
2. 标注哪些保持不动
3. 只对必要部分做专业化修正
4. 最终结果仍应保留原先的视觉记忆点与风格方向
```

## 适用场景

- 新 landing page
- 营销页改版
- 仪表盘视觉升级
- 组件库外观提升
- 用户反馈“太普通、太像模板”时

## 不建议这样用

- 一句话同时要求两个 skill “直接生成最终页面”
- 在没有风格方向、受众、参考物时就让它们直接出代码
- 让 `ui-ux-pro-max` 在第一轮主导整体设计方向
