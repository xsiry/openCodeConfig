# OpenCode 本地 AGENTS 约定

## 作用

- 本文件只保留当前用户长期有效的全局协作约束；运行时真相以 `~/.config/opencode/opencode.json[c]`、Oh-My-OpenAgent 配置、官方 schema 与 `doctor` 输出为准。
- 不在此重复 agent、category、model、MCP、plugin、hook、并发数、默认 provider 等动态事实；凡是可稳定写入配置、skill 或插件的内容，优先不要堆进 `AGENTS.md`。

## 硬约束（必须遵守）

- Thinking 使用中文；面向用户的回复默认使用简体中文，除非用户明确要求其他语言。
- 文件统一使用 UTF-8；除非文件本身已使用其他字符集，否则不要引入 GBK / ANSI。
- 优先使用官方来源；非官方镜像、第三方搬运站或付费分发站点不能作为规范依据。
- 优先复用仓库现有实现、命名、结构和约定；以最小安全改动完成任务，不为了“更优雅”主动偏离项目风格。
- 在引入新的第三方依赖、框架、MCP 或外部服务前，必须先检查项目依赖与配置；若需新增长期能力，必须先征得用户同意。
- 结论优先基于实际读取、搜索、诊断或验证结果，不对未读代码和未验证行为作判断；遇到错误时严禁盲猜修复，必须基于真实日志和局部代码读取闭环分析。
- 阅读代码坚持按需加载，先局部定位，再展开上下文，避免无意义的大范围整仓/整文件读取。
- 除非用户明确要求，严禁主动执行破坏性操作、修改 Git 历史、推送远程或直接提交；若需提交，必须先检查变更并给出 commit message 草稿。
- 验证 OpenCode 配置时优先查看官方 config docs / schema 与本地 `opencode.json[c]`；验证 Oh-My-OpenAgent 配置时，优先查看官方仓库 / release / schema 与本地 `oh-my-openagent.json[c]` 及兼容旧名 `oh-my-opencode.json[c]`。若新旧配置并存、命名迁移或生效路径有歧义，优先运行 `bunx oh-my-openagent doctor --verbose`（或兼容旧包名命令）确认实际生效配置。

## 代码注释硬约束

- 默认不为显而易见的代码补充注释，不得写成逐行翻译；但缺少必要解释会被视为实现不完整。
- 公共类、公共方法、对外暴露函数，以及关键代码片段，不得仅因“代码可读”而免于注释。
- 以下场景必须解释 `why / constraint / invariant / side effect`，而不是重复代码字面含义：
  - 公共类、公共类型、模块入口：职责、适用范围、使用限制。
  - 公共方法、对外函数、复用型 Hook：入参约束、返回值语义、主要副作用。
  - 关键业务规则：状态流转、金额/配额计算、权限判断、兼容逻辑、数据修正规则。
  - 非直观实现：顺序依赖、性能优化、缓存策略、降级逻辑、特殊边界处理、看似多余但实际必要的分支。
  - 外部约束：第三方 API 限制、协议字段约定、历史兼容原因、数据格式陷阱、平台差异。
  - 易误改节点：魔法值、时序要求、不可随意删除的保护逻辑、隐含不变量。
- 注释应放在关键语句或逻辑块上方，保持简洁、可维护；禁止“初始化数据”“处理结果”这类空洞注释。
- 修改关键逻辑时必须同步更新相关注释；结束实现前必须自检是否存在“看得出做什么，但看不出为什么这样做”的关键节点。

## 强建议（默认按此执行，除非有明确理由偏离）

### 证据、工具与检索

- 能从仓库内已有代码、配置、schema、注释和本地文档回答的问题，不要先求助外部资料。
- 默认优先本地结构化工具，其次本地文本搜索/读取；只有在本地证据不足且问题确实依赖外部事实时，再升级到官方文档、定向 MCP 或外部检索型子代理。
- 搜索范围必须显式收敛；避免对根目录、`node_modules` 或其他大目录做无过滤搜索。
- 已有足够证据支持结论时必须停止追加检索；连续两轮没有新增信息时，应转入汇总而不是继续扩搜。
- 长链路任务中优先保留“关键发现的简洁摘要”，而不是把大段原始输出反复塞进上下文。

### 委派、技能与外部能力

- 简单的多文件读取、精确搜索或局部替换，优先在当前会话完成，不要为了形式化而委派子代理。
- 需要委派时，必须给出明确范围、目标和产出，避免“随便看看”“探索一下”这类无边界指令。
- 区分 agent 与 category：category 是 `task()` 的任务预设，不是可递归协作的 agent。需要继续统筹、分派或多轮协作时，优先使用主编排 / 合适 agent，而不是把 category 当作可无限扩展的代理链。
- 子代理、category、后台任务并发与可用能力受当前 runtime 配置约束；不要在 `AGENTS.md` 中写死并发数字、模型分工或某个能力的启停状态。
- 当任务与某个 skill、插件或已启用能力明显匹配时，优先复用它；但若该规则更适合结构化配置、skill 文档或插件逻辑，就不要继续膨胀 `AGENTS.md`。
- Oracle 仅用于高成本且高价值的问题：架构权衡、两次以上失败后的调试、跨系统复杂问题；普通配置核对、常规代码阅读与轻量 review 不应默认升级到 Oracle。

### search-mode / analyze-mode

- 当用户显式进入 `[search-mode]` 或 `[analyze-mode]` 时，可提升检索与上下文收集强度；但仍必须遵守“本地证据优先、官方来源优先、避免无边界扩搜”的基本原则。

### 默认执行偏好

- 轻量任务优先单会话、本地工具与可验证的串行执行；仅当任务明显扩展为多步骤、高不确定性或跨模块问题时，再升级流程。
- 在运行 build / compile / test 等高成本验证前，先做低成本前置检查；若缺少依赖、命令、环境变量、权限或外部服务，应标记为环境阻塞并说明缺口。
- 同一验证命令若因同类原因失败，在未改变代码、配置、依赖或环境前不得重复运行；若全量验证成本显著偏高，应先说明成本、目的与范围，并请示是否执行。
- 项目上下文、现有代码、配置与已读文档足以支持下一步时，不要重复向用户确认。
- 需要用户确认时，优先一次性给出 2 到 3 个可选方案、影响与推荐项，避免拆成过多轮确认。
- 删除文件、大规模重构、修改 Git 历史、变更环境配置、调整 CI、数据库变更等高影响操作仍需先征得用户确认。

## 维护

- 仅在本地长期工作方式变化，或官方行为变更已影响协作策略时更新本文件。
- 若只是模型、agent、插件、category、hook、并发或其他运行时能力变化，优先修改 JSON 配置、skill 或插件，不要把动态快照回写进 `AGENTS.md`。
